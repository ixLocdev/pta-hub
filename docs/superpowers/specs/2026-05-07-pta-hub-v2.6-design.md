# PTA Knowledge Hub — v2.6 Design

**Date:** 2026-05-07
**Author:** Lucas Deichl (with Claude)
**Status:** Draft for review
**Scope:** Six new features split across two releases.

## Goal

Move the plugin from "good search" to "actively maintained, responsive to its community" by adding tools that (a) help editors keep content fresh, (b) close the loop between what visitors search for and what gets created, (c) remove rough edges around long entries, draft sharing, and offline distribution.

## Release split

| Release | Features | Why grouped |
|---|---|---|
| **v2.6.0** | Reading time + TOC, QR codes, stale-content reminders, public preview links | Smaller scope, lower risk, all admin-side or read-only front-end. Lets the simpler ones bake in production before shipping the heavier work. |
| **v2.6.1** | Suggest-a-topic form, Wizard related-entries panel | Each touches more surfaces (new CPT, new front-end form, AJAX in the Wizard JS). Worth a separate release so any regressions are isolated. |

## Shared scaffolding

Each feature is one new class file under `pta-knowledge-hub/includes/`, registered from `pta-knowledge-hub.php` after the existing classes. This matches the established one-class-per-concern convention.

**New post meta keys** (all on `pta_knowledge` posts):
- `ptk_last_reviewed` — ISO date string. Auto-set on publish/update; manually settable via row action.
- `ptk_preview_token` — 32-char random hex.
- `ptk_preview_expires` — Unix timestamp.
- `ptk_toc_cache` — serialized TOC HTML cached at save time.

**New options:**
- `ptk_review_threshold_months` — default 12. Exposed via filter `ptk_review_threshold_months` for future configurability; no UI in this release.
- `ptk_suggest_page_url` — optional. If unset, the footer CTA links to the search page with `#suggest`.

**New custom post types:**
- `ptk_suggestion` (v2.6.1 only) — private, not public-queryable, not synced across multisite.

**Multisite behaviour:**
- Review status is per-site. A Council push does not reset the subsite's "last reviewed" clock.
- Suggestions stay local to the site they were submitted on.
- Preview tokens stay local; they do not propagate to subsite copies.
- QR codes encode the permalink of the entry on the current site (Council and subsite copies have different QR codes pointing to their respective URLs).

## Feature 1: Stale-content review reminders (v2.6.0)

**File:** `includes/class-review-reminders.php` (`PTK_Review_Reminders`)

**Surface area:**
1. Admin list table for `pta_knowledge`: a "Last Reviewed" column, sortable, with a colored dot indicator (green = fresh, amber = approaching threshold, red = past threshold).
2. Row action "Mark as Reviewed" that stamps `ptk_last_reviewed = current_time('Y-m-d')` via a nonced admin-post handler.
3. Auto-stamp on `save_post_pta_knowledge` when the post transitions to `publish` (initial publish, manual update, wizard save).
4. Dashboard widget "Needs Review" listing entries past threshold, oldest first, capped at 20, with one-click "Mark Reviewed" links and "Edit" links.

**Threshold:** 12 months. Constant in code, filterable via `apply_filters('ptk_review_threshold_months', 12)`.

**Capability:** widget visible to `edit_posts` (consistent with v2.5.0 analytics change). "Mark Reviewed" requires `edit_post` on the specific post.

**Visibility states (color coding):**
- Green: ≤ threshold − 2 months
- Amber: within 2 months of threshold
- Red: past threshold (or never reviewed and `post_date` ≥ threshold ago)

## Feature 2: Public "Suggest a topic" form (v2.6.1)

**File:** `includes/class-suggestions.php` (`PTK_Suggestions`)

**Data model:**
- New CPT `ptk_suggestion`: private (`'public' => false`, `'show_ui' => true`, admin-only). Fields used: `post_title`, `post_content`, plus post meta `ptk_suggester_name`, `ptk_suggester_email` (optional), `ptk_suggester_ip_hash`.

**Front-end:**
- Shortcode `[ptk_suggest_form]` renders a form (title, body textarea, optional name, optional email, hidden honeypot field, nonce).
- Auto-injected footer CTA on every single `pta_knowledge` page: "Can't find what you need? Suggest a topic →" linking to either `ptk_suggest_page_url` (if set) or `#ptk-suggest` on the search page (the shortcode is expected to be embedded there).
- Submission via `admin-ajax.php` action `ptk_submit_suggestion` with: nonce check, honeypot check, IP-hash rate limit (3/hour using a transient `ptk_suggest_rate_<hash>`).
- Successful submission shows a thank-you message inline, no redirect.

**Admin:**
- New submenu "Suggestions" under PTA Knowledge.
- List table shows submission date, title, suggester (name or "anonymous"), preview of body.
- Row action "Convert to Draft" → creates a `pta_knowledge` draft pre-filled with title + body, links it to the suggestion via meta, then trashes the suggestion.
- Row action "Discard" → trashes the suggestion.

**Spam protection:**
- Honeypot field (hidden via CSS, bots fill it).
- IP-hash rate limit (no PII; SHA256 of REMOTE_ADDR).
- Empty submissions and submissions with URLs in the title rejected server-side.

## Feature 3: Per-entry QR codes (v2.6.0)

**File:** `includes/class-qr-codes.php` (`PTK_QR_Codes`)

**Library:** Bundle `chillerlan/php-qrcode` (MIT, ~30 KB, no runtime deps) under `pta-knowledge-hub/vendor/qrcode/`. Loaded only when the meta box renders or the download endpoint fires. No external API calls.

**UI:**
- Meta box "QR Code" in the post editor sidebar (priority `default`).
- Renders an inline SVG QR (~150×150 px) of the post's permalink.
- "Download PNG" button → admin-post action `ptk_download_qr` with `post_id` and a nonce. Returns a 600×600 PNG with `Content-Disposition: attachment; filename="pta-qr-<slug>.png"`.

**Capability:** meta box visible to anyone with `edit_post`. Download endpoint checks the same.

**No regeneration step:** the QR encodes the live permalink, so it stays valid as long as the slug doesn't change. We don't bother caching the SVG; it's cheap to regenerate.

## Feature 4: Reading time + auto Table of Contents (v2.6.0)

**File:** `includes/class-single-enhancements.php` (`PTK_Single_Enhancements`)

**Reading time:**
- Computed once per page load: `ceil(str_word_count(wp_strip_all_tags($post->post_content)) / 225)`.
- Inserted into the existing meta line in `templates/single-pta_knowledge.php` next to "Updated <date>".

**Table of Contents:**
- On `save_post_pta_knowledge`, parse `post_content` with `DOMDocument` to find all `<h2>` tags. Slugify text → assign `id` attribute. Build TOC HTML (a `<nav class="ptk-toc"><ol>...</ol></nav>` block).
- If `<h2>` count ≥ 3, store the rewritten content in a content filter (NOT in `post_content` — we don't want to mutate the saved post). The TOC HTML and the H2-id-rewritten content are cached together in post meta `ptk_toc_cache`.
- On front-end render, a `the_content` filter checks for `ptk_toc_cache` and, if present, prepends the TOC and uses the rewritten content.
- TOC links use `href="#<id>"`; smooth scroll handled in pure CSS via `scroll-behavior: smooth` on `html`.
- Mobile: TOC collapsed by default behind a `<details>` element. No JS required.

**Caching invalidation:** Cache is regenerated every save. Stored in post meta keyed to `post_modified` so a cache miss is detected if the post was updated outside our hook.

## Feature 5: Suggested related entries in the Wizard (v2.6.1)

**Wizard change:** add a side panel `<aside id="ptk-related-suggestions">` rendered next to the main form on the Wizard page (CSS grid; collapses below form on narrow screens).

**Trigger:** fires on `blur` of the title field, only after a category has been selected. Debounced 300ms in case of double-blur.

**Endpoint:** new `wp_ajax_ptk_wizard_related` AJAX action.
- Calls `PTK_Search_Engine::search($title, ['exclude' => $current_id, 'limit' => 3])`.
- Returns JSON: `[{id, title, permalink, excerpt}, ...]`.
- Capability: `edit_posts`.

**UI:**
- Empty state: "Suggested related entries will appear here once you fill in a category and title."
- Loading state: spinner.
- Results: card per entry with title, 1-line excerpt, "View" link (opens permalink in new tab), "Insert link" button.
- "Insert link" appends `[<title>](<permalink>)` to the description field's current value (the closest active textarea inside the form), with a leading newline if the field isn't empty.

**No new endpoint for non-Wizard editor.** The block editor sidebar can wait — the Wizard is where most volunteers create entries.

## Feature 6: Public preview links for drafts (v2.6.0)

**File:** `includes/class-public-preview.php` (`PTK_Public_Preview`)

**UI:** new section in the WordPress publish meta box (added via `post_submitbox_misc_actions`):
```
Share with reviewers
[ Status: No active link / Active until 2026-05-14 ]
[ Generate preview link ] [ Copy URL ] [ Revoke ]
```

**Token generation:**
- 32-char hex (`bin2hex(random_bytes(16))`).
- Stored as `ptk_preview_token` post meta.
- Expiry stored as `ptk_preview_expires` (current time + 7 days, Unix timestamp).
- "Generate" replaces any existing token (single active link per post).

**Front-end resolution:**
- `template_redirect` hook checks for `?ptk_preview=<token>` query var.
- If present:
  1. Look up the post by meta query.
  2. If post not found, expired, or already published → 404.
  3. Otherwise, set up the global $post, bypass `ptk_check_access` and `PTK_Role_Access`, and render the standard single template with a banner "Preview — not yet published. Anyone with this link can view this draft until <date>."

**Cleanup:**
- Daily WP-cron job (`ptk_cleanup_preview_tokens`) deletes meta where `ptk_preview_expires < time()`.
- Tokens auto-revoked when the post transitions from `draft`/`pending` to `publish`.

**Capability:** Generate/Revoke require `edit_post` on the specific post. The preview URL itself bypasses caps by design — that's the entire point.

## Build order

Within v2.6.0:
1. Reading time + TOC (front-end only, smallest surface)
2. QR codes (admin only, smallest surface)
3. Stale-content reminders (admin list table + dashboard widget)
4. Public preview links (template_redirect intercept, most subtle)

Within v2.6.1:
1. Suggest-a-topic form (CPT + shortcode + admin list)
2. Wizard related-entries panel (touches the Wizard JS, which is the most fragile area)

---

## Out of scope

- Configurable review thresholds with a UI — filter only for now.
- Multi-language support for new front-end strings — strings are wrapped in `__()` but the plugin has no `.pot` file yet; that's a separate effort.
- Email digests of stale content (deferred from #1 design — admin-only surface was requested).
- Block-editor sidebar version of the related-entries suggester (Wizard only).
- Bulk QR code "print sheet" page (deferred from #3 design).
- Single-use preview tokens (persistent within expiry was requested).
- Ratings / reasons on the helpful/not-helpful widget (out of v2.6 scope entirely).

## Verification plan

Each feature has a manual smoke test on the live multisite after deploy:

- **Reading time + TOC:** open an entry with 4+ H2s; confirm TOC appears, links jump correctly. Open a 1-H2 entry; confirm no TOC. Verify reading-time text shows.
- **QR codes:** open any entry's editor; confirm QR meta box renders SVG. Click "Download PNG"; confirm file downloads with correct filename and scans to the entry URL.
- **Stale content:** set `ptk_last_reviewed` to 14 months ago on one entry via DB or row action UI; confirm dashboard widget shows it; click "Mark Reviewed"; confirm it disappears from the widget.
- **Preview links:** create a draft; click "Generate preview link" in publish box; copy URL; open in incognito; confirm draft renders with banner. Publish the post; reload preview URL; confirm 404.
- **Suggest form:** submit one as anonymous, one with name+email, one with the honeypot filled. Confirm first two appear in admin, third is rejected. Convert one to draft; confirm draft has correct title/body.
- **Wizard related:** open Wizard, pick category, type a title that overlaps an existing entry. Confirm related panel populates. Click "Insert link"; confirm markdown link appended to description.

PHP lint clean is the floor; full verification is on-site.
